# 三网 TCP 延迟设计

## 目标与边界

在 VPS 详情页现有的“负载”“延迟”分段标签后增加“三网 TCP 延迟”。界面按省份行、联通/移动/电信三列展示，由当前 VPS 主动连接 93 个固定的 TCP-Ping 目标，视觉信息层级参考 ITDOG 的结果表，但保持 Naive Extended 的紧凑样式。

只有已登录管理员可以显式发起测试；访客不创建任务、不轮询、不调用管理员 RPC，只读取管理员最近一次保存的结果。功能不修改 Komari 后端、Agent、数据库结构、部署目录、反向代理或开放端口，也不引入运行时依赖。

本功能不是持续监控。管理员未点击测试时，宿主机和 Agent 不会因该功能产生网络连接、定时器、轮询或后台任务。

## 原生能力与方案选择

Komari 原生 `PingTask` 支持 `tcp` 类型；任务指定目标、客户端 UUID 和间隔后，由目标 VPS 的 Agent 执行 TCP 连接，结果可通过 `common:getRecords` 读取。现有的 `admin:addPingTask`、`admin:deletePingTask`、`admin:getAllPingTasks` 和 `common:getRecords` 足以完成一次测试及清理。

原生调度器没有“立即执行一次” RPC，且新增任务的首轮执行发生在间隔到期后。因此采用已确认的“临时原生 PingTask 批量测试 + 主题快照”方案：主题仅在管理员点击后短暂创建低间隔任务，收集首轮记录后写入紧凑快照并删除临时任务。

不采用以下方案：

- 浏览器直连目标：测得的是访客到目标的延迟，不能代表 VPS，且会把 93 个连接分摊给每位访客。
- 新增主题服务或外部 Worker：需要额外部署、密钥、数据库和维护面，超出主题范围。
- 修改 Komari 后端或 Agent：虽可增加真正的立即任务，但违反“不修改 Komari 后端和 Agent”的约束。
- 保留短间隔 PingTask：会变成持续监控，长期占用宿主机、Agent、记录库和网络。

## 固定目标目录

目标目录是编译进主题的静态只读数据，不提供管理员输入框、导入或 URL 覆盖。每个目标遵循：

```text
{provinceCode}-{carrierCode}-v4.ip.zstaticcdn.com:80
```

其中 `carrierCode` 只允许 `cu`（联通）、`cm`（移动）、`ct`（电信）。目录包含用户提供的 31 个省、自治区、直辖市及其三网组合，共 93 项，顺序固定为省份列表顺序内的联通、移动、电信。

固定目录既保证展示稳定，也避免管理员会话被主题利用为任意目标 TCP 连接的入口。运行时必须从目录的 host/port 重建任务，不信任 DOM、路由参数、快照或主题设置中携带的目标地址。

## 管理员测试流程

1. 管理员进入指定 VPS 的详情页，切换到“三网 TCP 延迟”，点击“开始三网 TCP 测试”。未登录访客只会看到只读结果；离线 VPS 的按钮禁用并说明不可测试。
2. 组件先调用 `admin:getAllPingTasks`，清理当前 VPS 名称空间中运行 ID 早于当前时间 5 分钟的遗留临时任务。运行 ID 是毫秒 Unix 时间戳；任务名使用稳定且可识别的 ASCII 前缀、节点 UUID、运行 ID 和目标序号，例如 `naive-tcp-v1-<uuid>-<epochMs>-<index>`。
3. 组件执行尽力而为的并发保护：同一页面同一节点只允许一个运行；扫描到任一仍在执行的本主题临时任务时拒绝新运行。没有后端锁时，多个独立管理员浏览器仍可能在极小竞争窗口同时发起测试，这是“不修改后端”的已知限制。
4. 主题按每批 24 项、批间串行创建 `type: "tcp"` 的 PingTask。每项只绑定当前节点 UUID，间隔为 5 秒。每批等待首轮间隔后，以有限次数读取该批任务的 `common:getRecords`，最多额外等待 12 秒；不使用无限轮询。
5. 每个任务只取本次运行窗口内最早一条有效记录。可用延迟四舍五入为非负整数毫秒；超时、无记录、异常值和创建失败统一记为 `null`。单项失败不会中断其余目标。
6. 所有批次结束后，主题将 93 个有序值、完成时间和快照版本保存到该节点的主题快照。保存成功后才将新结果显示为访客可见结果。
7. 无论成功、超时、页面卸载还是 RPC 失败，`finally` 都调用 `admin:deletePingTask` 清理本次实际创建的任务。清理失败的任务会在下次管理员测试的预清理阶段再次删除。

93 次 TCP 连接最多同时运行 24 项，每批最多约 17 秒，正常情况下测试约 1 分钟完成。运行期间组件显示确定的完成数、失败数与取消入口；路由离开、组件卸载或用户取消时立即停止创建后续批次，并尽力删除已创建任务。

## 结果快照与持久化

访客结果必须在管理员关闭页面后仍可见，因此不使用 `localStorage`、会话状态或原始 PingRecord 作为唯一来源。快照写入 Komari 已有的管理员主题设置保存通道，并通过已公开的 `common:getPublicInfo().theme_settings` 下发。主题不新增 HTTP 路由、RPC 方法、服务端文件或数据库表。

快照字段使用保留键 `threeNetworkTcpSnapshots`，值为 JSON 字符串；现有配置栏位不直接显示或编辑该内部字段。公开配置的其他键必须在保存时原样保留。数据契约：

```json
{
  "version": 1,
  "nodes": {
    "node-uuid": {
      "testedAt": "2026-07-15T12:34:56.000Z",
      "values": [23, 31, null]
    }
  }
}
```

`values` 的长度必须恰为 93，且严格对应固定目录。每项只能是 `null` 或 `0` 到 `60000` 的整数。解析时，未知版本、无效时间、未知节点键、错误长度、非有限数值或超范围数值会使该节点快照失效，而不是部分猜测或抛出异常。每次测试仅替换当前节点的快照，保留其他节点的最近结果。

实现前必须用当前目标 Komari 版本验证管理员主题设置保存通道能够由已登录主题页面调用、保持未知内部字段并将新值回传给 `common:getPublicInfo`。若该原生写入能力不存在或不满足这三个条件，则停止本功能实现并回到设计阶段；不得用自建接口、服务、文件写入或客户端持久化替代。

## 访客与管理员界面

标签始终位于详情页的现有“延迟”标签之后，使用异步组件加载，避免首页与非详情页加载 31 行结果表的代码。界面不使用 Canvas、ECharts、动画循环或嵌套卡片。

页面顶部显示本次结果的时间、来源节点和完成/失败数量。管理员额外看到开始按钮、进行进度、取消按钮和错误说明；访客只看到快照内容。无快照时，访客看到“管理员尚未完成测试”的静态空状态，管理员看到可执行的开始操作。

结果表在宽屏上采用一行一省份、三列运营商；单元格显示整数 `ms` 或失败标识。为便于扫描，延迟按阈值使用克制的绿色、琥珀色和红色状态色，失败采用中性色。窄屏改为每省份一个无边框分组，三项垂直排列，避免横向滚动和被截断文字。主题浅色/深色、背景模糊与无障碍焦点样式遵循现有 Naive UI 令牌。

## 模块边界

- `src/utils/threeNetworkTargets.ts`：静态的省份、运营商、目标键和展示信息；导出固定顺序，不含网络或 Vue 代码。
- `src/utils/threeNetworkSnapshot.ts`：纯函数，负责快照解析、合并、序列化、记录归属和延迟值校验。
- `src/utils/threeNetworkTcpTasks.ts`：仅负责管理员原生 RPC 的任务创建、有限等待、记录采集、取消与清理；不渲染 UI，也不解析主题设置。
- `src/stores/app.ts`：防御性读取 `threeNetworkTcpSnapshots`，只向组件暴露已归一化的节点快照；同时提供管理员主题设置读改写所需的受控入口。
- `src/components/ThreeNetworkTcpLatency.vue`：懒加载的显示与交互组件，管理员控制测试生命周期，访客仅消费 Store 数据。
- `src/views/InstanceDetail.vue`：仅注册异步组件并增加第三个分段标签。
- `src/utils/rpc.ts` 与 `src/utils/api.ts`：补充严格类型化的 PingTask 管理、记录和已验证的主题设置保存包装，禁止组件直接拼写 RPC 方法或参数。

## 安全、隐私与资源约束

- 管理员操作依赖 Komari 现有登录态和服务端管理员 RPC 授权；前端显示逻辑不是权限边界。
- 不允许自定义地址、端口、协议、客户端 UUID 或批量节点选择。目标、当前节点 UUID、类型和参数均在工具层校验。
- 不向第三方服务发送节点资料、结果、管理员凭据或访客资料。所有连接由 Komari Agent 到固定目标发起。
- 一次测试最多创建 93 个任务，实际并发最多 24，任务只存活测试窗口；没有长驻计时器、自动重试、定时重测或访客发起的探测。
- 访客加载时只解析其当前节点的一个最多 93 项快照，复杂度为 O(93)，无网络请求、无轮询、无任务创建。
- 主题快照只保存延迟/失败值和测试时间，不保存访问者 IP、探测过程日志、原始错误正文或管理员身份数据。

## 错误处理

- 管理员失去权限、节点离线、RPC 不可用或主题设置保存失败：显示可操作错误；不替换已有访客快照；始终清理已建任务。
- 单批创建或采集失败：对应目标记为 `null`，继续未完成批次；最终快照明确展示失败数量。
- 取消、路由切换、页面卸载：中止等待和后续创建，随后尽力清理；不保存部分结果，避免访客把中断测试误认为完整记录。
- 旧版本快照、配置被手动损坏或记录返回结构变化：安全降级为无结果，不影响详情页的负载和原有延迟图表。
- 下次管理员发起测试前的遗留任务清理发生失败：提示管理员，不创建新任务，防止叠加宿主机负载。

## 测试与验收

单元和合约测试至少覆盖：

- 31 个省份和三网目标的数量、顺序、固定 host/port 与不可变性。
- 快照的有效解析、错误版本、错误长度、非法延迟、损坏 JSON、节点级隔离与单节点合并。
- 任务按 24 项分批、无自定义目标、单项失败、首轮记录归属、12 秒上限、取消和 `finally` 清理。
- 管理员与访客的渲染边界：访客没有开始/取消控件，且不会调用管理员 RPC；无快照和失败快照的静态状态正确。
- 详情页标签顺序、异步加载、桌面三列和移动端纵向布局的组件合约。
- 主题设置读改写保留无关键，并且保存后的快照可经公共设置重新读取。
- 旧功能回归：原有负载、延迟图表、樱花、星轨、Live2D、剩余价值计算器和发布 ZIP 契约不受影响。

验收必须运行 `pnpm test:unit`、`pnpm lint` 和 `pnpm build`，并检查构建 ZIP 仍包含 `dist/`、`komari-theme.json` 与 `preview.png`，且未包含测试数据或管理员快照。
