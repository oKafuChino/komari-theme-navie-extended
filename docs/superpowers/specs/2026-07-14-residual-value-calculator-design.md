# Komari 剩余价值计算器设计

## 目标与边界

在公共探针页面 Header 右上角增加可选的剩余价值计算器。管理员启用后，访客点击计算器图标可在右侧抽屉查看所有 VPS 的剩余价值和目标币种总额。

功能完全运行在访客浏览器中，复用 Komari 已公开的节点数据，不修改 Komari 后端、API、数据库或服务进程。功能关闭时不执行计算、不请求汇率、不增加初始页面网络流量。

## 数据来源与计算

每个节点直接使用 Komari 数据中的 `price`、`billing_cycle`、`currency` 和 `expired_at`。剩余天数复用主题现有的 `getDaysUntilExpired(node.expired_at)`，因此与节点卡片显示保持一致，并按完整天数向下取整。

可计算节点必须同时满足：

- `price` 是大于零的有限数值。
- `billing_cycle` 是大于零的有限天数。
- `expired_at` 有效。
- 币种可识别。

单台节点的原币剩余价值：

```text
remainingDays = max(getDaysUntilExpired(expired_at), 0)
remainingValue = price * clamp(remainingDays / billing_cycle, 0, 1)
```

少于 24 小时的剩余时间按 0 天计算。结果最高不超过一期价格。已过期或免费节点价值为 0；一次性计费（`billing_cycle = -1`）、无效周期、无效日期、无效价格和未知币种仍显示在列表中，但标记原因且不计入总额。

## 币种与汇率

支持四种币种及固定符号映射：

| 显示符号 | 代码 | 含义 |
| --- | --- | --- |
| `¥` | `CNY` | 人民币 |
| `$` | `USD` | 美元 |
| `€` | `EUR` | 欧元 |
| `£` | `GBP` | 英镑 |

同时接受节点中对应的 `CNY`、`USD`、`EUR`、`GBP` 代码。`¥` 明确解释为人民币，不支持日元，避免符号歧义。

管理员可从 CNY、USD、EUR、GBP 中选择目标显示币种。备用汇率使用人民币为共同基准，表示“一单位源币种等于多少人民币”：

```json
{"CNY":1,"USD":7.2,"EUR":7.8,"GBP":9.2}
```

换算公式：

```text
targetValue = sourceValue * cnyPerSourceUnit / cnyPerTargetUnit
```

备用 JSON 只有在四个键均为正有限数值且 `CNY` 为 `1` 时有效，否则使用主题内置默认值。

## 在线汇率与缓存

访客首次打开抽屉时，先用管理员备用汇率立即计算，再由浏览器请求无需密钥且支持 CORS 的 Frankfurter 公开汇率 API：

```text
https://api.frankfurter.app/latest?from=CNY&to=USD,EUR,GBP
```

响应必须包含 `base: "CNY"` 以及 USD、EUR、GBP 三个正有限汇率。该接口返回“一人民币等于多少目标币种”，运行时将其归一化为备用配置使用的人民币基准：`cnyPerUnit[code] = 1 / response.rates[code]`，并设置 `CNY = 1`。成功响应校验后更新抽屉，并写入浏览器缓存键 `komari-naive-extended:residual-value:rates:v1`。

- 缓存有效期 12 小时。
- 请求超时 2.5 秒。
- 有效缓存优先于网络请求。
- 网络失败、超时、响应格式错误或存储不可用时继续使用备用汇率。
- 不自动重试，不轮询，不在抽屉关闭后持续工作。
- 请求只包含币种代码，不发送节点名称、价格、到期时间或其他 Komari 数据。
- 抽屉标明当前使用“在线汇率”“缓存汇率”或“备用汇率”，并在可用时显示更新时间。

汇率请求由访客浏览器直接发送，公开汇率服务会接触访客 IP。README 必须披露这一点。

## 管理员设置

在 `komari-theme.json` 的托管配置中增加：

| Key | 类型 | 默认值 | 用途 |
| --- | --- | --- | --- |
| `residualValueEnabled` | `switch` | `false` | 是否向访客显示计算器入口 |
| `residualValueCurrency` | `select` | `CNY` | 目标币种，选项为 `CNY,USD,EUR,GBP` |
| `residualValueFallbackRates` | `string` | `{"CNY":1,"USD":7.2,"EUR":7.8,"GBP":9.2}` | 人民币基准备用汇率 |

设置在 `src/stores/app.ts` 中统一进行防御性归一化，组件不得直接解析主题设置。

## 界面与交互

计算器图标位于 Header 右上角“自动主题/浅色主题/深色主题”按钮左侧，使用现有图标系统中的计算器图标、圆形图标按钮和 tooltip。功能关闭时该按钮不渲染。

点击按钮打开右侧抽屉：

- 桌面端使用紧凑固定宽度，移动端接近全宽。
- 顶部显示目标币种总额、参与计算节点数、未计入节点数、汇率来源和更新时间。
- 下方为单层紧凑列表，不使用图表或嵌套卡片。
- 每行显示节点名称、原生剩余天数、原价/计费周期、目标币种剩余价值或无法计算原因。
- 可计算节点按目标币种剩余价值从高到低排列，无法计算项放在末尾。
- 金额保留两位小数并使用目标币种符号。
- 加载在线汇率时抽屉保持可用，备用结果不会被 loading 遮挡。
- 键盘、焦点关闭、ESC 和遮罩关闭遵循 Naive UI Drawer 的标准行为。

## 模块边界

- `src/utils/residualValue.ts`：纯函数，负责设置校验、币种识别、单节点折算、汇率转换、排序与汇总。
- `src/utils/exchangeRates.ts`：负责在线汇率请求、2.5 秒超时、12 小时缓存、存储降级和响应校验。
- `src/components/ResidualValueCalculator.vue`：负责 Header 图标、抽屉状态和展示，不复制业务计算。
- `src/components/Header.vue`：只负责把计算器组件放在主题模式按钮左侧。
- `src/stores/app.ts`：归一化三个管理员设置。

不新增运行时依赖。继续使用 Vue、Naive UI、dayjs 和浏览器原生 `fetch`/存储 API。

## 错误处理

- 汇率 API 或缓存失败：使用备用汇率并显示来源，不影响页面其他功能。
- 单节点数据异常：仅隔离该节点，显示无法计算原因。
- 所有节点均不可计算：总额显示目标币种 `0.00`，并显示未计入数量。
- 功能运行时不得抛出未处理 Promise rejection。
- 抽屉关闭或组件卸载时中止未完成的汇率请求。

## 性能要求

- 功能关闭时零计算、零汇率请求。
- 功能开启但抽屉未打开时不请求汇率，不遍历节点计算价值。
- 每次节点或汇率变化只进行一次 O(n) 汇总。
- 不使用图表、Canvas、动画循环、轮询或新增第三方依赖。
- 12 小时缓存减少公开 API 请求；失败后不自动重试。

## 测试与验收

单元测试覆盖：

- 四种符号和代码映射。
- 复用向下取整剩余天数后的比例计算与一期价格上限。
- 已过期、免费、一次性、无效日期、无效周期、未知币种。
- 源币种到四种目标币种的换算和两位小数展示。
- 多节点排序、汇总、未计入计数。
- 备用汇率 JSON 校验。
- 有效缓存、过期缓存、存储不可用、API 成功、超时、错误响应和中止。
- 功能关闭时不渲染入口且不请求汇率。
- Header 中计算器图标位于主题按钮左侧。
- 抽屉的统计、来源状态、列表、空状态和移动端宽度合约。
